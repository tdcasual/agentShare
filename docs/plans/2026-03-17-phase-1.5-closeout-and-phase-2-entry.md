# Phase 1.5 Closeout And Phase 2 Entry Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Finish the remaining Phase 1.5 runtime and reliability gaps so the project has a trustworthy full-stack baseline before starting formal Phase 2 feature work.

**Architecture:** Treat this stage as a stabilization gate, not a feature sprint. Close the startup, schema, authz, and gateway semantics gaps first; then align the web console with the real API so Phase 2 can build on a verified control plane instead of a partially mocked one.

**Tech Stack:** FastAPI, SQLAlchemy, Redis, OpenBao, Next.js, Playwright, Docker Compose

---

### Task 1: Restore A Real End-To-End Startup Path

**Files:**
- Modify: `apps/api/app/main.py`
- Test: `apps/web/playwright.config.ts`
- Verify: `README.md`

**Why this comes first:**
- Current E2E is blocked at API startup, so every later stage would be built on unverified assumptions.

**Definition of done:**
- API boots cleanly under Uvicorn.
- Playwright can start the API and web app from config.
- README verification steps match the real startup flow.

### Task 2: Add Deterministic Schema Bootstrap For Fresh Environments

**Files:**
- Create: `apps/api/Dockerfile`
- Create: `apps/api/alembic.ini` or equivalent schema bootstrap entrypoint
- Create: `apps/api/alembic/env.py`
- Create: `apps/api/alembic/versions/001_initial_schema.py`
- Modify: `docker-compose.yml`
- Modify: `apps/api/app/db.py`

**Why this comes second:**
- The code has ORM models, but fresh runtime environments still lack a canonical way to create tables.
- Docker already expects an API image, but the build artifact is missing.

**Definition of done:**
- `docker compose up -d` can build and boot the API service.
- A new environment can create the required schema without test-only helpers.
- The API healthcheck passes after boot.

### Task 3: Enforce Real Authorization At Task And Capability Boundaries

**Files:**
- Modify: `apps/api/app/auth.py`
- Modify: `apps/api/app/routes/tasks.py`
- Modify: `apps/api/app/routes/invoke.py`
- Modify: `apps/api/app/routes/leases.py`
- Modify: `apps/api/app/services/task_service.py`
- Modify: `apps/api/app/services/gateway.py`
- Test: `apps/api/tests/test_tasks_api.py`
- Test: `apps/api/tests/test_invoke_api.py`
- Test: `apps/api/tests/test_lease_api.py`
- Create: `apps/api/tests/test_authorization_policy.py`

**Why this comes third:**
- Authentication exists, but policy enforcement is still thin.
- The design explicitly says tasks cannot grant privileges beyond what the agent already has.

**Definition of done:**
- Agents can only claim allowed task types.
- Agents can only invoke allowed capabilities.
- Lease issuance checks task scope, capability mode, and agent permission.

### Task 4: Replace Gateway Soft-Fallbacks With Explicit Runtime Behavior

**Files:**
- Modify: `apps/api/app/services/gateway.py`
- Modify: `apps/api/app/services/adapters/generic_http.py`
- Modify: `apps/api/app/services/adapters/openai_adapter.py`
- Test: `apps/api/tests/test_generic_http_adapter.py`
- Test: `apps/api/tests/test_openai_adapter.py`
- Create: `apps/api/tests/test_gateway_fail_closed.py`

**Why this comes fourth:**
- Current gateway behavior still masks adapter failures by returning echo-like success payloads.
- That is acceptable for scaffolding, but not for a trustworthy control plane baseline.

**Definition of done:**
- Adapter failures return clear API errors instead of fake success.
- Proxy results remain sanitized.
- Lease behavior is clearly documented as placeholder metadata or upgraded to a real lease artifact flow.

### Task 5: Align The Web Console With The Real Backend

**Files:**
- Modify: `apps/web/lib/api.ts`
- Modify: `apps/web/app/playbooks/page.tsx`
- Modify: `apps/web/app/actions.ts`
- Modify: `apps/web/app/agents/page.tsx`
- Modify: `apps/web/app/capabilities/page.tsx`
- Modify: `apps/web/app/tasks/page.tsx`
- Test: `apps/web/tests/*.spec.ts`

**Why this comes fifth:**
- The web UI currently remains resilient by falling back to mock data, but that can hide real integration failures.
- Before Phase 2, the console should surface backend truth rather than quietly substituting demos.

**Definition of done:**
- Playbooks load from `/api/playbooks/search`.
- API failures are visible in the UI.
- Fallback data is limited to intentional local-demo behavior, not silent production-path substitution.

### Task 6: Verify The Phase 1.5 Exit Gate

**Files:**
- Modify: `README.md`
- Verify: `docker-compose.yml`
- Verify: `apps/web/playwright.config.ts`

**Required verification:**
- `.venv/bin/pytest apps/api/tests -q`
- `cd apps/web && npm run build`
- `cd apps/web && npx playwright test`
- `docker compose up -d`
- `curl http://127.0.0.1:8000/healthz`

**Exit criteria:**
- End-to-end local verification passes from a clean environment.
- No route or service depends on `app.store` for runtime behavior.
- Docker and README reflect the actual developer workflow.

### Task 7: Begin Formal Phase 2 Only After The Exit Gate Passes

**Candidate Phase 2 streams from the original design:**
- MCP server exposure
- Richer playbook search
- More adapters
- Approval policies

**Recommended order inside Phase 2:**
1. Richer playbook search and real playbook UX
2. Approval policy model and enforcement
3. One additional first-class adapter beyond OpenAI
4. MCP server exposure on top of already-stable API operations

**Reasoning:**
- MCP exposure should come after API semantics are stable.
- Approval policies depend on authz and gateway behavior being correct.
- Better playbook retrieval is the lowest-risk user-visible Phase 2 increment.
