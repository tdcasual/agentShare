# Phase 1.5: Tech Debt Resolution Design

**Date:** 2026-03-16

## Overview

Phase 1 MVP is complete with all 10 planned tasks implemented. However, the MVP uses in-memory storage, hardcoded authentication, echo-only gateway proxy, and no concurrency protection. This phase resolves these technical debts before proceeding to Phase 2 features.

## Strategy

Bottom-up incremental replacement: Postgres → Redis → Auth → Gateway. Each step keeps the system runnable with passing tests.

## Scope

1. Postgres persistence (replace in-memory Store)
2. Redis task locks and idempotency keys
3. API Key authentication (replace hardcoded tokens)
4. Gateway real proxy with adapters (replace echo)

---

## 1. Postgres Persistence

### Current State

`apps/api/app/store.py` is an in-memory `Store` dataclass with Python dicts and auto-incrementing counters. All data is lost on restart.

### Design

- Introduce SQLAlchemy 2.0 (sync mode with `create_engine`) + Alembic for migrations. Use synchronous API to avoid converting all existing sync route handlers to async. Async migration can be done in a future phase if needed.
- Create ORM models for each core concept:
  - `Secret` (reference only, no plaintext — `store.secret_values` is NOT migrated to Postgres; secret plaintext remains exclusively in the secret backend (OpenBao/InMemory). `InMemorySecretBackend` retains its own internal dict for plaintext storage, independent of the old `Store` dataclass.)
  - `Capability`
  - `AgentIdentity`
  - `Task`
  - `Run`
  - `Playbook`
  - `AuditEvent`
- Model layering: ORM models for persistence (`app/models/`), Pydantic schemas for API request/response (`app/schemas/`), explicit mapping functions between them. Existing Pydantic models in `app/models/` are moved to `app/schemas/` and ORM models take their place.
- Create a Repository layer (one Repository class per model) encapsulating CRUD operations
- Routes obtain Repositories via FastAPI dependency injection (session-per-request pattern: one SQLAlchemy `Session` per request, committed on success, rolled back on error), replacing direct `store` access
- Retain `InMemoryStore` for test fallback; tests default to SQLite in-memory via SQLAlchemy
- Initial Alembic migration includes all tables. Rollback: each migration includes a `downgrade()` function.
- All tables include `created_at`, `updated_at` timestamps and appropriate indexes
- Column definitions match existing dict keys in `store.py`

### Dependencies

- `sqlalchemy>=2.0`
- `alembic`
- `psycopg2-binary` (Postgres sync driver)

### Infrastructure

- Add `api` service to `docker-compose.yml` with `depends_on` health checks for Postgres and Redis
- Add `healthcheck` directives to Postgres and Redis services
- Add startup connection retry logic (3 attempts, 2s backoff) for database and Redis connections

## 2. Redis Task Locks and Coordination

### Current State

Task claiming directly mutates in-memory dict status with no concurrency protection.

### Design

- Introduce `redis` client (sync), managed via FastAPI lifespan (connection pool)
- Task claim uses Redis distributed lock (`SET NX EX` pattern):
  - Lock key: `task_lock:{task_id}`, TTL 30 seconds
  - After acquiring lock: update task status in Postgres, then release lock
  - Lock acquisition failure returns 409 Conflict
- Idempotency keys for invoke and lease operations:
  - `Idempotency-Key` request header
  - Redis stores request result with TTL 5 minutes
  - Duplicate requests return cached result
- Tests use `fakeredis` as Redis substitute

### Dependencies

- `redis[hiredis]>=5.0`
- `fakeredis[lua]` (test)

## 3. API Key Authentication

### Current State

`auth.py` uses a hardcoded `KNOWN_AGENTS` dict for token-to-agent lookup.

### Design

- `AgentIdentity` table adds `api_key_hash` column (SHA-256 hash, never stores plaintext)
- Web console adds Agent management page:
  - Create Agent generates random API Key (32-byte hex)
  - Plaintext key shown only once at creation time
  - List/revoke existing agents
- Auth flow: `Authorization: Bearer <api_key>` → SHA-256 hash → query Postgres → return `AgentIdentity`
- Key revocation: Web console marks key as `revoked`, auth check rejects revoked keys
- Bootstrap: `BOOTSTRAP_AGENT_KEY` environment variable for initial setup and testing. On startup, if the env var is set and no agent with that key hash exists, the system auto-creates a bootstrap agent with `name=bootstrap`, `status=active`, and full capability access. This agent is used for initial configuration and test seeding.
- Test fixtures inject test Agent and Key via Repository

### Security

- API keys are 32-byte cryptographically random hex strings
- Only SHA-256 hash stored in database
- Revoked keys are rejected immediately
- Rate limiting on auth failures (future consideration, not in this phase)

## 4. Gateway Real Proxy

### Current State

`gateway.py` `proxy_invoke` echoes parameters instead of calling external APIs.

### Design

#### Secret Value Flow in Gateway

After the Store is replaced, the gateway obtains secret values through the `SecretBackend` interface (not the Repository). Flow: Gateway receives `capability_id` → queries Capability Repository for `secret_id` and `adapter_config` → calls `SecretBackend.read_secret(secret_id)` to get plaintext → passes to adapter. The Repository never touches secret plaintext.

#### Adapter Interface

```python
class CapabilityAdapter(Protocol):
    async def invoke(self, secret_value: str, params: dict) -> dict: ...
```

#### Adapters

1. **GenericHttpAdapter**: Universal HTTP proxy
   - Capability config specifies `target_url`, `method`, `headers_template`
   - Gateway injects secret into designated position (e.g., `Authorization: Bearer {secret}`)
   - Forwards request and returns response

2. **OpenAIAdapter**: Extends generic adapter
   - Pre-configured for OpenAI Chat Completions endpoint
   - Handles streaming and non-streaming responses
   - Response body trimming (removes unnecessary large fields)

#### Adapter Registry

- `capability.adapter_type` field determines which adapter to use
- Default: `generic_http`
- Capability model adds fields: `adapter_type`, `adapter_config` (JSON)

#### Safety Measures

- Target URL allowlist (specified in Capability config, domain-level)
- Response body size limit (default 1MB)
- Request timeout (default 30 seconds)
- Secret values redacted from response logs and audit events

### Dependencies

- `httpx` (already present)

## Testing Strategy

- Unit tests: Repository CRUD, lock acquisition/release, key hashing, adapter request building
- Integration tests: SQLAlchemy against SQLite, fakeredis for locks, httpx mock for adapters
- API tests: All existing tests migrated to use real DB session (SQLite)
- E2E tests: Playwright tests updated for new auth flow

## Migration Path

- Existing in-memory tests continue to work during transition (SQLite backend)
- No breaking API contract changes — all endpoints keep the same request/response shapes
- `docker-compose.yml` already declares Postgres and Redis services; will add API service with health check dependencies
- New environment variables: `BOOTSTRAP_AGENT_KEY`, `DATABASE_URL`, `REDIS_URL` (already in config)

## Resolved Decisions

- Use synchronous SQLAlchemy (not async) to avoid disrupting all existing sync route handlers
- Secret plaintext stays exclusively in SecretBackend, never in Postgres
- Model layering: ORM models (persistence) + Pydantic schemas (API) with explicit mapping
- Bootstrap agent auto-created on startup from env var
- Gateway obtains secrets via SecretBackend interface, not Repository

## Open Questions

- Whether OpenAI adapter should support streaming proxy responses in v1
- Exact rate limiting strategy for auth failures (deferred to Phase 2)
